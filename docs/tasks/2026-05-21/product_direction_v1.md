# 山野气象台 / 路线决策台：产品方向备忘

日期：2026-05-21

## 当前定名

- 气象工具：山野气象台
- 路线工具：路线决策台

这两个名字暂定。后续可以微调，但当前页面、任务书、数据结构先围绕这两个方向推进。

## 产品重点

山野气象台不是普通天气页。它服务小地点、山口、营地、村庄、路线起点和坐标查询。重点是定位准确、气象指标专业、手机端阅读清楚。

路线决策台不是普通地图页。它服务徒步、骑行、登山三类主流户外活动。重点是轨迹可信度、补给、撤退、风险、装备、季节和天气窗口。

## 主入口

只保留三类主入口：

- 徒步
- 骑行 / 摩旅
- 登山

暂时移除“其他户外”主入口。原来的其他数据可以留档，但不进入主导航。以后如果要做露营、观星、桨板、越野跑、亲子户外、自驾穿越，需要单独建立“活动场景库”，并重新收集资料。

## 轨迹分级

每条线路必须标明轨迹等级：

- 实走轨迹：有可下载或可解析的 GPX、KML、GeoJSON。
- 来源轨迹：有外部轨迹页，但本站不能解析或下载。
- 关键点示意：只有起点、终点、营地、垭口、补给点、撤退点，用直线连接。
- 待核验：缺少可靠地理点，不画线或只显示区域。

页面必须明确提示：关键点示意不可直接导航。

## 页面当前必须改的问题

- 山野气象台版式偏弱，字体不够专业，需要重做手机端视觉。
- 路线决策台的数字分数不能裸露给用户。改为资料等级、轨迹等级、风险标签。
- 小地点定位要增加候选确认、置信度、行政区、经纬度来源。
- 国外线路要补，但必须归入徒步、骑行、登山，不做笼统的“国外户外”。
- 真实轨迹能接就接，接不了就标为来源轨迹或关键点示意。

## 后续数据原则

先搭框架，再逐批补数据。不要一次把杂乱资料塞进页面。每批资料都要有来源、可信度、字段完整性和用途说明。

## 国外板块

路线决策台必须加入国外板块。国外不是第四种运动类型，而是线路区域筛选。

页面结构建议：

- 第一层：徒步 / 骑行 / 登山
- 第二层：国内 / 国外 / 全部

国外线路仍然必须归入徒步、骑行或登山。不要建立“国外户外”或“其他国外”。

国外线路进入页面前，必须标明：

- 国家或地区
- 运动类型
- 轨迹等级
- 数据来源
- 是否有真实 GPX/KML/GeoJSON
- 是否只是关键点示意

## 版式要求

山野气象台的版式可以推翻重做。当前版本只作为功能原型，不作为最终视觉方向。

新版气象页必须满足：

- 手机端优先。
- 字体要有专业仪表感，不要像普通宋体或默认黑体网页。
- 首屏必须清楚：查哪里、定位准不准、当前风险是什么。
- 信息密度要高，但不能乱。
- 视觉要像专业气象工具，不像普通天气 App。
- 如果现有结构限制视觉效果，可以推翻重做。

路线决策台当前版式可以保留方向，但需要继续提高信息表达：

- 裸数字分数改成明确标签。
- 轨迹等级必须可见。
- 国内 / 国外筛选必须清楚。
- 手机端点击线路后，地图和详情都要看得到。
